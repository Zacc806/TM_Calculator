# Верификация точности расчётов калькулятора

**Объект:** `src/core/calc.ts` → `computePayment` на `calculator.atamuragroup.kz`
**ТЗ Задача 3.** Допустимое расхождение с банком: **< 1%**.
**Дата автономной верификации:** 2026-06-08.

## 1. Методология

Калькулятор считает **аннуитетный** платёж по стандартной формуле, которую в РК используют все банки
для аннуитетных займов (то же, что Excel `=PMT`):

```
Платёж = K · i · (1+i)^n / ((1+i)^n − 1),    i = годовая_ставка / 12 / 100,   n = срок в месяцах
K = стоимость − первоначальный взнос
```

Для беспроцентной рассрочки (ставка 0%) — равные доли `K / n` без переплаты.
Платёж округляется до целого ₸; «общая сумма» и «переплата» выводятся из округлённого платежа,
поэтому три цифры на карточке всегда согласованы.

**Проверка** — двумя независимыми путями:
1. **Юнит-тесты** (`src/core/calc.test.ts`, блок *bank parity*): для каждого кейса платёж движка
   сверяется с независимо посчитанным эталоном PMT — допуск ±1 ₸ (округление) и < 1%.
2. **Харнесс** (`scripts/calc-verify.ts`, запуск `npx tsx scripts/calc-verify.ts`): прогон
   представительных кейсов с выводом таблицы ниже.

## 3. Важно: номинальная ставка vs ГЭСВ

Калькулятор использует и показывает **номинальную договорную ставку** (например, 7% / 9% / 20,5%) —
именно она определяет аннуитетный платёж по договору.

**ГЭСВ** (годовая эффективная ставка вознаграждения) — это раскрытие, включающее комиссии и страховку,
и она **законно выше** номинальной. Поэтому при формальной сверке с банком:
- «ежемесячный платёж», «общая сумма», «переплата» должны совпасть с банком < 1% (чистый аннуитет);
- строка «реальная (эффективная) ставка / ГЭСВ» у банка будет ВЫШЕ номинальной в калькуляторе —
  **это не ошибка формулы**, а разные метрики. Калькулятор сознательно показывает номинальную ставку
  (как в договоре), а не ГЭСВ.

Дополнительные источники легитимных расхождений в пределах 1%: разное округление у банка, страховые/
комиссионные платежи, для Отбасы — гибридная схема «накопление + заём» по части программ.
